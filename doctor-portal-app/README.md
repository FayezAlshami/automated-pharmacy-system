# doctor-portal-app

بوابة الطبيب ضمن مشروع "الصيدلية المؤتمتة". هذا التطبيق مستقل بالكامل عن تطبيق المريض وتطبيق الأدمن/الصيدلي، ومبني كواجهة Front-End فقط حالياً مع Mock Data منظمة، لكنه مهيأ معمارياً للانتقال لاحقاً إلى Back-End حقيقي دون إعادة هيكلة جذرية.

## 1) الهدف من المشروع

هذا التطبيق يخدم رحلة الطبيب داخل المنصة من أول زيارة وحتى:

- فهم فكرة النظام عبر صفحة تعريفية حديثة.
- تسجيل الدخول أو إنشاء حساب جديد.
- الوصول إلى لوحة رئيسية مختصرة وسريعة.
- تنفيذ بحث عام عبر كامل الكتالوج.
- تصفح الأقسام الدوائية.
- استعراض تفاصيل أي دواء.
- حفظ أدوية داخل المفضلة.
- إضافة أدوية إلى السلة.
- مراجعة آخر الطلبات عبر صفحة مستقلة باسم "سجلات الطلبات".

المشروع لا ينفذ Back-End فعلي ولا Checkout حقيقي ولا وصفات حقيقية حالياً، لكنه مبني على طبقة خدمات تجعل هذا الانتقال لاحقاً مباشراً نسبياً.

## 2) أبرز ما تم تنفيذه في النسخة الحالية

- إعادة بناء واجهة الطبيب بصرياً لتبدو أقرب إلى منتج فعلي وليس Demo أكاديمي.
- دعم لغتين داخل التطبيق: العربية والإنجليزية.
- زر تبديل لغة واضح يعمل على مستوى التجربة كاملة.
- صفحة تسجيل/إنشاء حساب محسنة بصرياً.
- حقل التخصص في إنشاء الحساب أصبح اختياراً من متعدد بدلاً من Text حر.
- إضافة تأكيد كلمة المرور داخل التسجيل.
- إزالة صفحة "ملف الطبيب" من الرحلة الحالية.
- استبدالها بصفحة مستقلة باسم "سجلات الطلبات".
- تفعيل زر المفضلة فعلياً.
- إضافة صفحة مخصصة للمفضلة.
- إزالة قسم آخر الطلبات من لوحة الطبيب الرئيسية.
- إعادة ترتيب بطاقات الوصول السريع لتظهر بشكل شاقولي وأكثر وضوحاً.
- استبدال أيقونات أقسام الأدوية بأيقونات أحدث وأكثر تعبيراً عن المجال الطبي.

  للتحقق من المدخلات وفرض قواعد واضحة مثل صحة البريد الإلكتروني وتأكيد كلمة المرور.

## 7) متغيرات البيئة

مثال الملف موجود في:

- `.env.example`

المتغيرات الأساسية:

```env
VITE_APP_NAME=بوابة الطبيب - الصيدلية المؤتمتة
VITE_API_BASE_URL=http://localhost:3000/api
VITE_USE_MOCK=true
```

شرحها:

- `VITE_APP_NAME`
  اسم التطبيق الظاهر في الواجهة. حالياً الواجهة نفسها تستخدم نصوصاً ثنائية اللغة داخل الكود، لذلك هذا المتغير ليس مسؤولاً وحده عن الترجمة.

- `VITE_API_BASE_URL`
  نقطة البداية لأي API حقيقي عند الربط الفعلي.

- `VITE_USE_MOCK`
  المفتاح الذي يحدد هل التطبيق يعمل على بيانات Mock أو على API حقيقي.

## 9) شرح بنية المجلدات عملياً

- `components/common`
  مكونات عامة قابلة لإعادة الاستخدام مثل `SearchBar` و`LanguageSwitch`.

- `components/features`
  مكونات مرتبطة بمجال الأدوية نفسه مثل `CategoryCard` و`DrugCard`.

- `components/layout`
  الغلاف التشغيلي العام بعد تسجيل الدخول، وأهم ملف هنا هو `DoctorAppShell.tsx`.

- `config`
  تهيئة البيئة العامة وتهيئة Axios وقراءة متغيرات البيئة.

- `constants`
  ثوابت غير متغيرة مثل بيانات التخصصات والأوصاف ثنائية اللغة والـ metadata الخاصة بالأقسام الدوائية.

- `hooks`
  منطق قابل لإعادة الاستخدام مثل `useDrugFilters` و`usePortalLocale`.

- `mocks/data`
  بيانات الوهم المنظمة للتجربة الحالية: أطباء، شركات، أدوية، طلبات.

- `pages`
  الصفحات الفعلية المعروضة للمستخدم.

- `routes`
  تعريف المسارات العامة والمحميّة.

- `services`
  طبقة الوصول إلى البيانات سواء من Mock أو من API حقيقي.

- `store`
  الحالة المركزية المشتركة داخل البوابة.

- `types`
  العقود البرمجية الأساسية التي تتعامل معها المكونات والخدمات.

- `utils`
  أدوات مساعدة مثل تنسيق التاريخ والعملة والتخزين المحلي.

## 19) كيف يتم استدعاء بيانات الـ Mock؟

التدفق الحالي:

1. الصفحة تطلب بيانات من `service`.
2. الـ service يقرأ `appConfig.useMock`.
3. إذا كانت القيمة `true`:
   يتم إرجاع البيانات من `src/mocks/data`.
4. إذا كانت القيمة `false`:
   يتم تنفيذ طلب حقيقي عبر Axios.
5. الصفحة لا تعرف مصدر البيانات الحقيقي، وتتلقى فقط نفس النوع البرمجي.

## 20) كيف نلغي الـ Mocking لاحقاً؟

الخطوات العملية:

1. عدّل `.env` واجعل:

```env
VITE_USE_MOCK=false
```

2. اضبط:

```env
VITE_API_BASE_URL=https://your-real-api.example.com
```

3. حدّث endpoints داخل:

- `src/services/authService.ts`
- `src/services/catalogService.ts`
- `src/services/orderService.ts`

4. أبقِ الصفحات والمكونات كما هي طالما أن الـ services ترجع نفس الـ types الحالية.

## 23) كيف نحافظ على نفس الـ Interfaces أثناء النقل من Mock إلى API؟

القاعدة المتبعة في هذا المشروع:

- الأنواع الموجودة في `src/types/doctor-portal.ts` هي العقد الداخلي للتطبيق.
- أي Response مختلف من الخادم الحقيقي يجب تحويله داخل `services`.
- يمنع أن تتعامل الصفحات مباشرة مع DTO خام من الـ API.

مثال مفاهيمي:

- إذا أعاد الخادم `drug_name` بدلاً من `name`
- فإن التحويل يجب أن يحدث داخل `catalogService`
- وليس داخل `DrugCard` أو `CategoryPage`

## 24) شرح تدفق البيانات داخل المشروع

التدفق المعتاد:

1. المستخدم يتفاعل مع الصفحة.
2. الصفحة تستدعي خدمة أو Action من الحالة المركزية.
3. إذا كان الحدث يخص الجلسة أو السلة أو اللغة أو المفضلة، يمر عبر Zustand.
4. إذا كان الحدث يخص الكتالوج أو الطلبات، يمر عبر `services`.
5. الـ services تختار Mock أو Axios.
6. البيانات تعود إلى الصفحة بنفس الشكل البرمجي المتوقع.

## 25) كيف نضيف صفحة جديدة؟

1. أضف ملف الصفحة داخل `src/pages`.
2. إذا احتجت مكونات جديدة قابلة لإعادة الاستخدام:
   ضعها داخل `components/common` أو `components/features`.
3. أضف route جديداً داخل `src/routes/AppRouter.tsx`.
4. إذا احتجت بيانات جديدة:
   أضف type مناسباً، ثم Mock Data، ثم method داخل service مناسب.

## 26) كيف نضيف endpoint حقيقي جديد؟

مثال: إضافة Endpoint للمفضلة لاحقاً.

1. أضف أو حدّث النوع المطلوب في `src/types`.
2. أنشئ method جديدة داخل `catalogService` أو خدمة مستقلة مثل `favoriteService`.
3. أضف تنفيذ Mock مقابل لها عند الحاجة.
4. أضف تنفيذ Axios لنفس الـ method.
5. احرص أن يرجع النوع نفسه في الحالتين.

## 27) شرح بعض الملفات الأساسية مفاهيمياً

- `src/routes/AppRouter.tsx`
  نقطة دخول المسارات العامة والمحميّة.

- `src/components/layout/DoctorAppShell.tsx`

  أكبر ملف بيانات وهمية ويغذي معظم تجربة الكتالوج.

## 28) قرارات معمارية مهمة

- تم إبقاء الترجمة خفيفة عبر Hook داخلي بدلاً من مكتبة i18n كاملة لأن المشروع ما زال Front-End فقط، مع بقاء إمكانية التوسع لاحقاً.
- تم نقل "آخر الطلبات" إلى صفحة مستقلة حتى تصبح لوحة الطبيب أكثر هدوءاً وتركيزاً.
- تم تفعيل المفضلة على مستوى الحالة المركزية وليس داخل صفحة واحدة.
- تم حفظ اللغة والمفضلة في Local Storage لضمان استمرار التفضيلات بين الزيارات.
- تم إبقاء Checkout شكلياً كما طلبت، لكن مع واجهة قابلة للتحويل لاحقاً إلى Order Flow حقيقي.

## 29) افتراضات ونقاط مؤجلة

- لا يوجد Back-End حقيقي حتى الآن.
- لا يوجد Checkout فعلي.
- لا توجد وصفات حقيقية أو موافقات تأمينية.
- المفضلة محفوظة محلياً فقط حالياً.
- السلة غير مرتبطة بخادم بعد.
- الترجمة الحالية تغطي الواجهة الأساسية بشكل عملي، لكنها ليست نظام i18n مؤسساً على ملفات ترجمة خارجية بعد.
